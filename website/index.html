Turning overlooked free-text fields into a reviewable taxonomy and analyzable labels.

Med Taxonomizer is an open research toolkit for teams with existing free-text medical data that rarely enters quantitative analysis. It helps convert heterogeneous notes, comments, and registry text fields into a transparent taxonomy and reproducible per-record labels that can be joined back to the dataset for statistical analysis.

Why this is hard

Free-text fields contain analyzable information, but usually remain outside the dataset.

Many clinical registries and research datasets include short free-text fields for comments, complications, adverse events, or unexpected findings. These fields often contain the most specific information, but they are difficult to use because labels are inconsistent, multi-event descriptions are common, and no fixed taxonomy exists up front.

Method   What the toolkit does

The workflow starts with an existing corpus of free-text records. Text is normalized, non-informative entries are filtered, and the remaining descriptions are processed in batches so that even short, messy, multi-label clinical notes can become structured research variables.

Instead of asking users to hand-design a label space, the taxonomy builder proposes domains and categories bottom-up. Multiple model runs are consolidated into a stable draft taxonomy, then reviewed, pruned, and exported as a versioned tree.

In the taxonomy-tree workflow, an open-weight model such as Qwen 3.5 27B can induce a multi-level taxonomy directly from the corpus. The point is not that one fixed taxonomy is universal, but that the same process can be rerun for another dataset and clinical domain.

Once the taxonomy is fixed, the labeler applies the same schema to the full corpus. A small human-reviewed sample can be used as a quality check before scaling, but the main deliverable is a statistically analyzable label table for the whole dataset.

End-to-end pipeline

From existing free text to a taxonomy tree and full-dataset labels.

Each stage writes inspectable artifacts: taxonomy candidates, consolidation decisions, run metadata, and final per-record labels. A team can start with its own text column and end with variables that are ready for statistics.

Stage I

Taxonomy induction

Multi-seed LLM ensemble discovers domains and categories bottom-up from de-identified text. No predefined label space.

  • 6 seeds × domain discovery, then per-domain category ensembles
  • LLM-guided consolidation merges seeds into a flat taxonomy
  • Split audit + subcategory expansion produces the final tree
Stage II

Review & taxonomy stabilization

Human review makes the discovered tree editable and auditable. Reviewers can rename, merge, split, or approve labels before the taxonomy is frozen for full-corpus labeling.

  • Path-aware search across the evolving tree
  • Approve, rename, merge, split, or mark uncertain nodes
  • Optional reviewed sample to check stability before scaling
Stage III

Full-dataset labeling

The selected labeling configuration is applied to every eligible text record, producing one or more validated taxonomy labels per entry.

  • Async, resumable SQLite-backed batches (size 25, concurrency 6)
  • Exports tidy tables for statistical analysis, dashboards, or manual review
  • Keeps taxonomy version, prompt file, and model metadata attached

Corpus-derived taxonomy discovery

The tree is not predefined. It is induced from the corpus, consolidated, pruned, and exported for analysis.

Med Taxonomizer treats taxonomy construction as a staged scientific process: discover broad meta-domains, consolidate them, discover categories inside each domain, consolidate again, then expand over-broad branches into subcategories and leaves.

Tree construction from corpus to export

Each stage asks a narrower taxonomy question, then merges overlapping proposals before the next layer is built. Over-broad nodes are expanded or pruned before the final export is used for labeling.

Three tools, one workflow

From an empty taxonomy to labeled records, reusable for another corpus.

The toolkit publishes the core capabilities a research team needs to apply this method: inductive taxonomy generation, optional human review of the tree, and full-corpus labeling that scales to a complete dataset.

Tool 01 · taxonomy induction

Generate a multi-level taxonomy from free-text data.

A staged Python pipeline that can run with an open-weight LLM such as qwen/qwen3.5-27b. It discovers domains, categories, and subcategories without requiring a predefined label space. Intermediate JSON artifacts preserve the taxonomy decisions at every step.

Qwen 3.5 27B model-agnostic Pydantic schemas structured JSON
  1. Multi-seed meta & category discovery (6 + 3 seeds)
  2. LLM-guided consolidation, per-domain & cross-domain
  3. Flat labeling of all entries with multi-label support
  4. Constrained other review for missing categories
  5. Three-seed split audit, subcategory expansion, tree pruning
Public capability: build a multi-level taxonomy tree from a text column.
Tool 02 · taxonomy review

Local reviewer to stabilize the taxonomy before labeling.

The repository includes installable local reviewer code. Teams can load a taxonomy JSON file in their own browser, record approve, rename, merge, split, or reject decisions, and export a review CSV.

local browser app taxonomy review CSV versioned exports audit notes
  1. Run with cd apps/reviewer && python3 -m http.server 5173
  2. Load a taxonomy JSON file locally in the browser
  3. Capture reviewer decisions and notes for ambiguous nodes
  4. Export a spreadsheet-friendly review queue
  5. No hosted service or bundled project dataset is required
Public capability: create a stable taxonomy release before full-corpus labeling.
Tool 03 · full-dataset labeling

Full-corpus labeler for the entire dataset.

Applies the selected prompt and taxonomy to all eligible records with batched, resumable structured output. The result is a tidy label table that can feed statistics, dashboards, or downstream models.

asyncio SQLite cache batch 25 / conc. 6 resumable
  1. Loads the final taxonomy tree & applies the selected prompt
  2. Constrains output to valid taxonomy paths via JSON schema
  3. Writes one row per record × concept or project-specific grouping
  4. Skips IDs already present, so reruns continue without rework
  5. Exports labels for joining back to your source table
Public capability: export joinable labels for downstream analysis.

Lightweight validation

A small quality check before full-dataset labeling.

Validation is intentionally lightweight: review a sample, check that the taxonomy can be applied reliably, then label the full dataset. Provenance or source-field analyses should be treated as context and discordance checks, not as gold-standard accuracy unless separately adjudicated.

Optional reviewed-sample check

A reviewed sample can help choose a labeling configuration before applying it to every record. Quantitative scores should only be reported when a genuine human reference set exists.

Documentation & reproducibility

Every artifact, prompt, and run is inspectable.

Documentation

Bring your own corpus

Start from a CSV, Parquet, or database extract with a text column. The docs walk through taxonomy induction, review, validation, and full-corpus labeling.

Models

Open-weight by default

Discovery can run with qwen/qwen3.5-27b or another structured-output-capable model. Proprietary models are optional comparison points, not a requirement for using the tool.

Determinism

Schema-validated outputs

Every model response is parsed against a Pydantic JSON schema with two-shot repair retries; invalid outputs are rejected, not silently corrected.

Traceability

Inspectable run metadata

Each phase records the model, seed, prompt version, and export metadata needed to audit the taxonomy and labeling outputs end-to-end.

Release

Open-source toolkit release.

The public repository contains the toolkit code, prompts, documentation, synthetic examples, and the local reviewer. Cite the repository version or commit used for your analysis.
